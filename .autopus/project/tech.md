### GPT/Codex Context Admission

- 800–2,000-token receipt budget은 memory/knowledge/index의 선택적 recall에만 적용한다. core/SPEC 필수 문서는 별도 frozen layer이며 요약·절단·누락 대상이 아니다.
- `auto workflow context`는 core/SPEC와 존재하는 architecture 문서를 전체 읽고 secret redaction 및 prompt-injection neutralization 뒤 전달한다. body-free receipt에는 변환 전 raw `source_hash`와 실제 전달된 sanitized `prompt_hash`, complete 상태, manifest hash를 함께 기록한다. `auto workflow binding`은 root, command, SPEC, 현재 파일 hash와 supervisor의 `--context-required-document` 집합을 재검증해 omission/replay를 거부하며 실패 시 canonical `full_ultra`를 유지한다.
- retained GPT/Codex worker는 최종 worktree 배정 후 snapshot을 한 번 만들고 direct 호출과 phase-split의 모든 phase에서 같은 frozen snapshot을 재사용한다. planner 이후 각 phase에는 원 태스크 계약도 한 번씩 재첨부하며 두 층 모두 compressor 밖에 둔다. 필수-context 검증 실패나 128K 초과는 provider 호출 전에 차단한다.
- worker가 반환하도록 요청하는 `context_ack`는 진단 신호이며 enforcement gate가 아니다. 강제 계약은 supervisor-held expected ref 집합의 완전 일치와 source/prompt/manifest hash 검증이다.
- `auto spec review`의 complete admission은 configured reviewer가 모두 `codex`/`openai`/`gpt` alias일 때만 켜진다. 각 리비전은 `--required-document`와 `--conditional-profile`을 포함한 supervisor-held 옵션으로 `BuildContextDelivery`를 실행한 뒤 같은 옵션으로 엄격하게 검증하고, core, 존재하는 architecture, 추가 필수 문서와 SPEC 4종 전문을 중복 없이 provider prompt에 주입한다. 문서 누락, receipt 변조, ref 집합 불일치, stale snapshot, 잘못된 SPEC identity 또는 128K 초과는 orchestra/provider 호출 전에 차단한다. mixed/Claude/Gemini review는 기존 excerpt prompt를 유지하며, `pkg/pipeline`의 provider-neutral receipt helper에는 현재 non-GPT production caller가 없다.
- required layer의 secret은 redaction하고 prompt-injection 표식은 neutralize하되 raw/prompt 해시와 invalidation metadata로 변환 사실을 보존한다. 이 구현은 Go 표준 라이브러리만 사용하며 Claude/Gemini 런타임 경로를 확대하지 않는다.
